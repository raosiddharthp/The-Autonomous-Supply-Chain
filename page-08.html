01 — GCP Reference Architecture

Two regions.
One jurisdiction.

EU data residency is not a configuration toggle — it is a Terraform constraint enforced at the organisation policy level. All workloads, all data, all encryption keys remain within the EU at all times. Frankfurt anchors the system. Amsterdam holds the recovery posture.

PRIMARY REGION
europe-west3 · Frankfurt
All production workloads · All MDR & ISO 13485 device records
MDR/GDPR EU data residency enforced. All PII and device records confined to europe-west3. Org Policy constraint active.
ISO 13485 Device history records, batch records, and NCR audit trails written and stored exclusively within this region.
CMEK All data encrypted with customer-managed keys. Key ring meddevice-asc-keyring in europe-west3. Google cannot access data at rest.
DR REGION
europe-west4 · Netherlands
Disaster recovery · Async replication · EU jurisdiction maintained

Analytics RPO: 4 hours · RTO: 8 hours — reporting and analytics modules. No patient safety or compliance impact during extended window.

02 — Compute Architecture

Stateful agents.
Stateless inference.

Agent workloads are inherently stateful and long-running — they belong on GKE with namespace isolation and Workload Identity. ML inference modules are stateless and spiky — Cloud Run with per-module service accounts and zero min-instances for batch. The SAP integration surface lives on Apigee X. Three runtimes. Subscription Patterns
Push Subscriptions — Real-Time Agent Triggers
Push subscriptions deliver events directly to Cloud Run endpoints, triggering agent modules within the latency SLO. No polling. No queue depth accumulation under normal load.
SLO: ≤60s end-to-end latency for real-time topics
Pull Subscriptions — Batch Analytics Consumers
BigQuery sink consumers and reporting pipelines use pull subscriptions. Designed for throughput over latency — they batch-read at scheduled intervals during off-peak windows.
Scheduled: 22:00–06:00 CET for ML training and bulk analytics jobs
Dead Letter Topics — All Failures Captured
Every subscription has a corresponding dead letter topic. No message is silently dropped. Failed deliveries after maximum retry exhaustion are captured, alarted, and available for replay. Failure rate is a P2 alert condition.
Dead letter per subscription · P2 alert on DLT accumulation
Message Retention
All topics retain messages for 7 days. This provides a replay window for any consumer that falls behind or needs to reprocess events during an incident.
7 days retention · All topics uniform policy

BigQuery Access Control
COL Column-level security on all supplier PII fields in asc_supplier. Procurement analyst roles see aggregated risk scores. Raw PII requires explicit data steward approval.
ROW Row-level security by manufacturing site across inventory and quality datasets. Site managers see only their site's data. No cross-site visibility without explicit admin grant.
AUD asc_audit: write-only service account. The audit log writer can only append. No principal — human or service account — holds delete permission on asc_audit. Immutability is enforced at the IAM layer, not by convention.
/agent_state/{'{'}agentId{'}'}/{'{'}sessionId{'}'}
Current agent execution state. Read/write by the owning agent service account only. Cross-agent state access is not permitted — agents communicate via Pub/Sub.
/hitl_queue/{'{'}checkpointId{'}'}
Pending HITL decisions. Written by agent, read by approval interface, updated by reviewer action. Timeout behaviour triggers escalation path after configured window.
/a2a_transactions/{'{'}transactionId{'}'}
A2A Commerce loop state. Tracks the full negotiation lifecycle: offer → counter → accept/reject. Each state transition is atomic and logged to asc_audit.
/supplier_risk/{'{'}supplierId{'}'}
Latest supplier risk score plus history of score changes. Written by SupplierSentinel agent. Read by ProcureGuard for purchase decision context.
05 — Security Architecture

Zero trust.
No exceptions.

BeyondCorp for all human access. Workload Identity for all service-to-service calls. VPC-SC perimeter makes data exfiltration structurally impossible, not just policy-prohibited. CMEK means Google cannot access ClaraVis data at rest — not by accident, not by legal compulsion.

VPC Service Controls Perimeter
SC VPC-SC perimeter: meddevice-asc-prod. Resources inside: BigQuery, Firestore, Vertex AI, Cloud Storage, Secret Manager, Pub/Sub. No resource inside the perimeter is reachable from outside except through defined access levels.
AL Access levels: Corporate network + BeyondCorp device policy. Both must be satisfied simultaneously. A corporate network connection from an unmanaged device does not satisfy the access level.
EG Ingress/egress rules enforce that only Apigee X is permitted to call external APIs (SAP Ariba, Salesforce, Veeva Vault). No other service inside the perimeter can make external API calls.
Identity & Access Control
BC BeyondCorp Enterprise — all human access to GCP Console and HITL approval interfaces. Context-aware access. Device trust required. No VPN. No implicit perimeter trust.
WI Workload Identity Federation — all service-to-service calls. No service account key files generated or stored anywhere. Short-lived credentials only.
SM Secret Manager — all API keys, CMEK key references, and connection strings. Zero secrets in code, environment variables, or container images. Automatic rotation enforced: external API keys (SAP Ariba, Salesforce, Veeva Vault) rotate every 90 days via Secret Manager rotation notifications → Cloud Functions rotation handler. Rotation events are logged to asc_audit. Any secret version older than 90 days without a rotation event triggers a P2 alert.
IAM Binding Example — SupplierSentinel SA
supplier-sentinel-sa →
BigQuery.dataViewer on asc_supplier
Pubsub.publisher on asc.hitl.queue
Pubsub.subscriber on asc.supplier.events
Nothing else.
CMEK — Customer-Managed Encryption Keys · Key Ring: meddevice-asc-keyring (europe-west3)
BigQuery All 8 datasets encrypted with CMEK. Default table encryption overridden at dataset level in Terraform.
Firestore All 4 collections CMEK-encrypted. Agent state, HITL queue, A2A transactions, supplier risk scores.
Cloud Storage All GCS buckets CMEK-encrypted. Model artifacts, training data, Terraform state bucket.
Vertex AI Model training jobs, Feature Store, and Vertex AI Pipelines use CMEK via the vertexai-key key. Training data processed in-memory never leaves the CMEK-protected storage layer.
Pub/Sub All 10 topics encrypted at rest with CMEK. Message payloads containing supplier PII or device event data are covered. CMEK enforced at topic creation in Terraform — not applied retroactively.
Secret Manager All secrets encrypted with CMEK. Secret versions are immutable; access grants are logged to Cloud Audit Logs and sink to asc_audit.
MedDevice holds the encryption keys. Google cannot access data at rest through operational access or legal compulsion directed at Google. Note: CMEK does not protect against legal compulsion directed at MedDevice as the key holder. Key access governance and legal hold procedures are addressed in the Data Protection Impact Assessment.
06 — Terraform IaC Structure

Infrastructure as
code. No drift.

Every GCP resource in this architecture is defined in Terraform. Eight modules. Three state files. Nightly drift detection. Any unplanned change to production infrastructure triggers a P1 alert before it causes a compliance violation.

Module Structure · /modules/*
/modules/networking VPC, VPC-SC perimeter, firewall rules, Private Service Connect
/modules/compute GKE Autopilot cluster, Cloud Run services, node pool definitions
/modules/data BigQuery datasets, Firestore collections, GCS buckets, access controls
/modules/messaging Pub/Sub topics, push/pull subscriptions, dead letter topics, retention policies
/modules/security IAM bindings, Secret Manager secrets, CMEK key rings, VPC-SC access levels
/modules/ml Vertex AI Feature Store, Model Registry, Pipeline definitions, endpoint configs
/modules/integration Apigee X proxies, API product configs, rate limit policies, circuit breakers
/modules/observability Cloud Monitoring dashboards, alerting policies, log sinks, SLO definitions
State Management
Remote State — GCS Bucket · Inside VPC-SC Perimeter
Terraform state stored in meddevice-asc-tf-state GCS bucket. CMEK-encrypted. Versioned. No local state files. The state bucket is inside the VPC-SC perimeter — it contains a complete infrastructure map (resource IDs, IAM bindings, secret references) and must be treated as sensitive. A dedicated VPC-SC Access Level grants the Cloud Build service account (cloudbuild-sa@meddevice-asc-shared.iam) perimeter ingress solely for Terraform plan/apply operations. No other principal or pipeline may access the state bucket from outside the perimeter.
State Locking — GCS Native + Firestore
State locking via GCS object versioning + Firestore lock table (europe-west3). Terraform Cloud is explicitly avoided — it is a US-hosted SaaS product that would receive infrastructure metadata and create a GDPR data residency question. All locking state remains inside the EU org boundary. Concurrent apply prevention is enforced without any dependency on a non-EU third party.
Environment Isolation
Separate state files per environment: prod / staging / dev. A staging deploy cannot modify prod state under any circumstance.
Drift Detection
Terraform plan runs nightly via Cloud Build. Any diff between Terraform state and actual GCP resource configuration triggers a P2 alert. Unresolved drift for 24 hours escalates to P1 — it is a potential compliance event.
07 — CI/CD Pipeline

Eight stages.
One named approver.

Cloud Build orchestrates every deployment. Feature branches deploy to dev automatically. Production requires a named human approver. Integration tests validate cross-agent A2A flows before any container is built. The EU AI Act compliance check is not optional — no ML deployment reaches production without a validated Model Card. Binary Authorization ensures only attested images reach GKE and Cloud Run.

TRIGGER 01 · Push to feature branch
Automatic deploy to dev environment. All stages run. No manual gate.
TRIGGER 02 · PR merge to main
Automatic deploy to staging + full integration test suite.
TRIGGER 03 · Manual approval
Deploy to production. Named approver required. Audit record written on approval.

09 — FinOps + GreenOps

Cost as code.
Carbon as signal.

Every GCP resource carries a cost label that maps to a module, environment, and cost centre. Batch ML training jobs run between 22:00 and 06:00 CET — when the grid is clean. ScopeTracer calculates its own GCP carbon footprint and includes it in the Scope 3 report it generates.

€
FinOps
Cost visibility by module, by environment, by cost centre
Cost labels on all resources: module · environment · cost_centre (per MedDevice department) · eu_ai_act_classification. Every GCP invoice line maps to a business owner. Label taxonomy (eu_ai_act_classification enum): high-risk-annex-iii · limited-risk · minimal-risk · non-ai-infra. Terraform enforces the enum — invalid values fail the plan stage. Labels map directly to EU AI Act Annex III classifications defined in the TOGAF Phase B register.
BigQuery slot reservations per environment prevent ad-hoc query cost spikes from analytics consumers. Query cost alerts at €500/day — before the problem is visible in the monthly bill.
Cloud Run min-instances=0 for all batch and reporting modules during off-peak. No idle compute for workloads that run on a schedule.
Monthly FinOps review: cost per module mapped against value delivered. Modules with poor cost-to-value ratios are candidates for architectural revision, not just optimisation.
◈
GreenOps
Carbon scheduling + Scope 3 self-reporting
Carbon-aware batch ML scheduling via the Carbon-Aware SDK. Training jobs are submitted to Cloud Scheduler with a carbon-intensity signal from the Google Carbon-Aware Batch Scheduling API (europe-west3). The 22:00–06:00 CET window is a fallback default only — the primary scheduling signal is real-time grid carbon intensity data, not a hardcoded time heuristic. This ensures training runs on clean energy by measurement, not assumption.
ScopeTracer reports its own GCP carbon footprint as part of the Scope 3 calculation it generates for MedDevice. The tool that measures supply chain emissions accounts for its own.
Cloud Carbon Footprint API integrated into the SupplyChain Command Dashboard. Carbon intensity for GCP workloads is a first-class metric alongside cost and latency — visible to the same leadership audience as FinOps figures.
↺ The loop closes here: ScopeTracer calculates supplier Scope 3 emissions, reports them to MedDevice's CSRD dashboard, and accounts for its own GCP compute carbon footprint in the same calculation. Infrastructure carbon is not an exemption. It is line one of the Scope 3 calculation.
