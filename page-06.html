Formal state machine for SupplierSentinel — the worked example that demonstrates the full HITL + A2A trigger pattern. Every state has a defined entry condition, action, exit condition, audit write, and timeout behaviour. All nine agents share this structural contract. Per-agent variations (timeouts, HITL IDs, escalation targets) are fully enumerated in the HITL Registry above.

SupplierSentinel was selected as the worked example because it demonstrates the complete pattern: continuous autonomous monitoring, SHAP-attributed risk scoring, HITL-gated escalation, and A2A Commerce trigger as the downstream action post-approval. The "0 Unaudited A2A Messages" guarantee is enforced at the tool layer: a2a_commerce_publisher invokes audit_log_writer as a pre-condition. The publish call will not execute if the Firestore write does not return a confirmed document ID. This is an SDK-level enforcement, not a policy convention.

Section 04

A2A Protocol Specification

Two distinct A2A patterns. Intra-enterprise coordination within MedDevice's GCP estate. Inter-enterprise A2A Commerce across the boundary to supplier and 3PL agent networks — the portfolio's primary technical differentiator, with full cross-boundary trust model and GDPR data minimisation specification.

Pattern 01 Intra-Enterprise A2A — Within MedDevice GCP Estate
Protocol
Orchestrator publishes to named agent topics. Agents subscribe and process. Responses published to reply topics. All messages schema-validated Protobuf with explicit field definitions below. All state written to Firestore before the agent proceeds to the next action. No agent-to-agent message is acknowledged without a prior Firestore record.
Transport Guarantees
Pub/Sub: exactly-once delivery (ordering keys)
Idempotency key: per-message UUID (v4)
Dead-letter: agent-dlq topic, 3× retry
Schema: Protobuf v3, strict version pinning
Auth: Workload Identity per agent service account
State persistence: Firestore before proceed
Audit: immutable write before every branch

Flow 01
DemandIQ → InventoryOrchestrator
Completed forecast output from DemandIQ feeds directly into the reorder model input for InventoryOrchestrator. Forecast results published to agent-inventory topic. InventoryOrchestrator subscribes and triggers ML reorder run. Closes the demand-to-replenishment loop without human intervention below €100K threshold. Duplicate message_id → idempotent discard.
Flow 02
SupplierSentinel → ProcureGuard
High-risk supplier event (post HITL-02 approval) triggers ProcureGuard to open a procurement review in SAP Ariba and initiate A2A SourcingRequest broadcast to pre-qualified alternative supplier agents. The HITL approval JWT is propagated forward through the A2A message chain. Supplier contact personal data is excluded from intra-enterprise messages — GDPR Article 5(1)(c) data minimisation.
Flow 03
QualityTrace → SupplierSentinel
An NCR event with identified component batch triggers SupplierSentinel re-evaluation of the affected supplier. QualityTrace publishes the component batch ID and NCR reference to agent-supplier topic. SupplierSentinel re-scores with NCR context included as a feature, elevating the risk signal with full regulatory traceability. MDR SLA timer runs independently of this flow.

Section 05

Cross-Boundary Agent Trust Model

The inter-enterprise A2A Commerce pattern crosses GCP project and organisational boundaries. This section specifies how agent identity is established, verified, and revoked — and how GDPR Article 46 cross-border transfer obligations are satisfied at the contract layer.

Identity Layer
Workload Identity Federation — Mutual
Both MedDevice and each supplier/3PL register a WIF identity before any A2A message exchange. MedDevice's ProcureGuard agent authenticates to external supplier agents using its GCP Service Account WIF token. Each supplier agent presents its own WIF token, which ProcureGuard verifies against the pre-registered Agent Card before accepting any BidResponse or ACK. No message from an unregistered WIF identity is processed. WIF tokens are short-lived (1 hour) and automatically rotated. Revocation is immediate via WIF pool membership removal.
Discovery Layer
A2A Agent Card Registry
Each supplier and 3PL agent publishes an Agent Card to a GCP-hosted A2A Agent Card registry at onboarding. The Agent Card contains: WIF identity, A2A endpoint URL, supported message types, ISO 13485 certification status, and GDPR data processing agreement reference. Agent Cards are signed by MedDevice's Procurement Director (HITL-06) at onboarding and re-validated annually. Revoked or expired Agent Cards immediately block that supplier from participating in A2A Commerce loops. ProcureGuard queries the registry before every SourcingRequest broadcast — no cached identity lists.
Message Integrity Layer
ECDSA P-256 Bi-directional Signing
Every A2A message — in both directions — is signed with ECDSA P-256. The sender's public key is retrieved from the Agent Card registry at verification time. This prevents: (1) message spoofing from unauthenticated agents, (2) man-in-the-middle bid injection, (3) replay attacks (per-message UUID checked against Firestore deduplication log). Signature verification failure → message rejected + security alert to CISO + Cloud Monitoring event. Signing keys rotated every 90 days via automated Cloud KMS workflow.
GDPR Transfer Mechanism
GDPR Art. 46 — Standard Contractual Clauses
Supplier agents outside the EEA: data transfer is covered by EU Standard Contractual Clauses (SCCs, 2021 Commission Decision) embedded in every supplier onboarding agreement. SCC reference is included in each supplier's Agent Card metadata. GDPR Article 30 Records of Processing: each Firestore audit document constitutes the processing record for the corresponding A2A transaction. Data minimisation (Art. 5(1)(c)): A2A message schemas are defined to exclude personal data of supplier contacts — supplierId fields contain WIF machine identities only.
Onboarding & Offboarding
Supplier Agent Lifecycle
Onboarding: supplier submits Agent Card + ISO 13485 certificate + DPA. Legal Counsel (HITL-06) approves registration. Agent Card signed and added to registry. WIF pool membership granted. Test A2A handshake performed before production access. Offboarding: WIF pool membership revoked immediately. Agent Card marked REVOKED in registry. In-flight A2A loops → circuit breaker to manual SAP Ariba workflow. All prior Firestore audit records retained per ISO 13485 retention schedule (15 years).
Security Incident Response
Compromised Agent Response
If a supplier agent's WIF identity is suspected compromised: immediate WIF pool revocation. All pending A2A Commerce loops for that supplier → circuit breaker state. CISO + Procurement Director alerted. Firestore audit log reviewed for anomalous messages. ECDSA signing key for MedDevice rotated as precaution. Any POs in PENDING state escalated to HITL-06 for manual review before issuance. Full incident record written to Firestore with immutable timestamp.

Architecture Position

The swarm that knows its limits.

The circuit breaker layer is not defensive programming. In a regulated medical device supply chain, a supplier risk score that silently fails is more dangerous than one that loudly escalates. Every circuit breaker in this architecture has a defined human escalation target, a defined fallback state, and an immutable audit record that captures the degraded-mode operation. The Orchestrator — previously unspecified — now carries the most critical circuit breaker definition in the swarm: the dead-letter queue, the ops runbook trigger, and the COO alert that ensures no message is silently dropped when the routing layer fails.

The MDR SLA hard escalation at hour 48 is not configurable. It is encoded as a write-path constraint. No agent state, no system outage, and no circuit breaker can suppress it. The CSRD HITL-07 gate is not configurable either — no regulatory report is filed without human sign-off, regardless of deadline pressure. This is what compliance as a structural property means in practice.
